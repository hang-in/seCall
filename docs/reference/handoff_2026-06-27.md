## 5. 작업 방식 (반드시 지킬 것)

- **fork PR 머지 전 로컬 풀 검증**: fork PR 은 CI 자동 실행 안 됨. `cargo fmt --check` + `clippy --workspace --all-targets -- -D warnings` + `cargo test --workspace` 직접. (memory `feedback_local_ci_gate`, `feedback_fork_pr_merge_gate`)
- **검증과 commit/push 분리 배치**: 같은 Bash 배치에 넣으면 검증 실패해도 push 됨. (memory `feedback_verify_before_push_separate_batch`)
- **release 흐름**: PR(Cargo.toml bump + CHANGELOG) → 머지 → `git tag vX` push → `release.yml` 자동 트리거. 수동 gh release create 불필요.
- **windows CI stall**: clippy step 20분+ 멈추면 runner stall → `gh run cancel` 후 `gh run rerun`.
- destructive(branch -D / 파일 삭제 / 되돌리기 어려운 것)는 confirm. push/PR/머지는 위임받음.
- **bash 출력 토큰 마스킹 주의**: rg/grep stdout 에서 vendor/model 문자열(예: `default_backend`→`n`, gemini)이 마스킹될 수 있음. 문자열 리터럴 확인·편집은 **Read 도구로 원문** 확인.
- **config.toml 에 API 키 평문 보관** (`~/Library/Application Support/secall/config.toml` line 24/38/43). 편집 시 키 라인 건드리지 말고 응답에 노출 금지.
